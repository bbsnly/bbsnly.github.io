Most rewrites serve the engineer, not the business

June 30, 2026 · 5 min read

For a few weeks one summer I woke at 4am to rewrite code my employer had already paid someone to write. It ran in production every day, and it was built on CakePHP. I barely knew CakePHP, so every file looked wrong to me. I knew Laravel, and I loved working in it. So on my own time, unasked, I ported the thing piece by piece, swapping in Illuminate packages until the old framework was gone.

We merged it. We started the next project in Laravel too. And I never once made the case for why it was worth doing, because it wasn't, at least not to the business. The application did the same job for the same users at the same speed as before. The rewrite improved exactly one thing: how the code felt to me.

That is the pattern worth naming. Most rewrites answer to the engineer - what they want to learn, what offends their taste, what looks good in an interview - and not to the company paying the salary.

Working code is a ledger of bugs someone already fixed

The strongest case against a rewrite is not the effort. It is memory. Code that has run in production for years is a written record of every bug someone hit and quietly fixed. The odd conditional, the retry with the suspicious timeout: most of it is scar tissue, and each scar marks an incident you never saw. Joel Spolsky made this point in 2000, calling the full rewrite the single worst strategic mistake a software company can make.

Throw the code away and you throw away the fixes. Then you meet the same bugs again, in production, in front of the same users.

I have worked on a Perl system that was already more than ten years old, and a homegrown PHP CMS of about the same age. Neither looked anything like what I would write today. Both just ran. The companies never reinvested in them, and that was the right call. The code had paid for itself long ago, and every quiet year it kept serving traffic was pure return.

Unfamiliar is not the same as broken

Here is the tell I missed in my own story. I was sure the CakePHP code was wrong, but I had never actually learned CakePHP. Unfamiliar is not the same as broken. When you don't know a tool well, everything built with it looks like a mistake, because you can't yet see the reasons behind the choices. That feeling of "this is all wrong" is often just the sound of you not understanding it yet.

Start from the opposite assumption: the people before you were not fools. They made choices that fit the use case and the constraints they had at the time. The code is shaped the way it is for a reason, even when that reason is no longer visible to you.

When touching it is the right call

None of this means freeze everything. That is the lazy reading of "if it ain't broke," and it is how you end up running an end-of-life runtime in 2026 with no security patches. Some debt really does come due. The honest forcing functions look something like this - not a complete list, just the ones I keep running into:

  • The runtime or a key dependency is end-of-life, with open CVEs and no upgrade path.
  • One person understands the system, and they just gave notice.
  • Every new feature costs three times what it should because of the design, and you can show the trend.
  • The business needs a capability the current code was never shaped to grow into.

That last one is where I am right now. I am rewriting a service, with an AI coding agent doing much of the typing, because we need it to do something it was never built for. The code works. The business outgrew it. That is a reason with a number behind it.

The test that sorts a real rewrite from a vanity one is that simple: can you put a figure on the pain? A CVE, a velocity tax, a hire you can't make, a launch you'll miss. If no number shows up, what you have is a preference about style.

AI makes a rewrite cheap to type and just as hard to get right

AI shifts this math, though not the way most people claim. The new argument is that rewrites are cheap again: the model can port a whole module in an afternoon, so why not let it. But typing was never the expensive part of a rewrite. Rediscovery was.

An agent generates the replacement in minutes. It does not know why the old code carried that strange timeout, because the reason lives in a Slack thread from 2021 and a postmortem nobody linked from the source. So it produces something clean and plausible, with the scar tissue stripped out. You ship it, and you meet the old bugs again, this time without the context the first author had when they fixed them. Fast generation of code that has forgotten its own fixes is not a discount.

The clean version will not stay loved either. The reason today's code reads like a pile of someone else's mistakes is that it survived contact with reality. The AI-written replacement earns the same fate. In two years an engineer opens it, can't see why it is shaped that way, and feels the identical pull to rewrite it "properly."

Cheaper generation doesn't end that loop. It makes the loop easier to start and harder to stop. The way out is not a slower model. It is writing the reasons down in plain language, so the next person, or the next model, gets the why, not just the what.

So before you rewrite something that works, go looking for the number. If you can't find one, the thing that needs maintenance is your patience, not the code.
